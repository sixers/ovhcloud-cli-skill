# ovhcloud-config

## When To Use

- The task is about `ovhcloud config ...`, profile management, endpoint selection, or default project resolution.
- You need to inspect the active profile or explain how configuration is resolved from file and environment.
- You need to document a config-changing command without actually applying it yet.

## Start Here

- Read [ovhcloud-shared](../shared/SKILL.md) first.
- Open the [generated config reference](../../references/generated/config.md) for the `config profile`, `config set`, and `config show` subcommands.
- Prefer `ovhcloud config show` and `ovhcloud config profile show` before any config-changing verb.

## Read-Only First

- `ovhcloud config show` and `ovhcloud config profile show` are the safest inspection entry points.
- `config set`, `config set-endpoint`, `config profile switch`, and `config profile delete` change local state and require explicit user approval.
- When documenting multi-profile flows, keep examples generic and do not overwrite the active profile during validation.

## Important Identifiers And Flags

- Profiles are stored as `[profile:<name>]` blocks in `~/.ovh.conf`.
- The active profile can come from `--profile`, `OVH_PROFILE`, or the `[default]` config section.
- `default_cloud_project` is profile-scoped, so account context matters when interpreting Public Cloud commands.
